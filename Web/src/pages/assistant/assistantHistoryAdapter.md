# Assistant History Adapter

## Goal

Transform `AiCallContext` data from the repository into a view model that the new `Assistant History` UI can render in user mode and developer mode.

## Files

- `assistantHistory.types.ts`
  Defines the public types used by the UI:
  - `AssistantHistoryConversation`
  - `AssistantTimelineEntry`
  - `AssistantToolKind`

- `assistantHistoryAdapter.ts`
  Main orchestration:
  - reads `AiCallContext`
  - builds the normalized timeline
  - derives title, preview and summaries

- `assistantHistoryToolMetadata.ts`
  Contains the tool interpretation rules:
  - maps tool names to `toolKind`
  - builds user-facing `title` and `description`
  - builds chips and developer previews

## Data flow

1. `buildAssistantHistoryConversations(contexts)`
   Filters compatible contexts and sorts them by most recent update.

2. `buildAssistantHistoryConversation(context)`
   Creates the final object used by list and detail screens.

3. `buildTimelineEntries(context)`
   Walks through `context.history` and converts each raw message into a normalized timeline entry.

4. `buildToolTimelineEntry(...)`
   Joins the `assistant.tool_calls` metadata with the corresponding `tool` result by `tool_call_id`.

5. `describeTool(...)`
   Applies the de/para rules for each known tool and returns:
   - `toolKind`
   - `title`
   - `description`

## Rendering contract

- `type: "system" | "user" | "assistant"`
  Rendered as message cards.

- `type: "tool"`
  Rendered as action cards.
  The icon is chosen from `toolKind` in the layout layer.

## Why there is a low-signal filter

The title and preview should prefer useful user prompts over noise like:
- metadata messages
- ultra short replies
- fragments with too little context

The current rule is not “greeting detection” anymore.
It is a low-signal heuristic based on message length and number of words, so the adapter avoids picking bad titles such as `oi`, `ok` or `mas ele`.

## i18n rule

All user-facing strings generated by the adapter must come from `Lang.visual.assistant.adapter`.

The adapter can still pass through raw data values such as:
- tool names
- ids
- domains not mapped yet
- routes not mapped yet

Those are data, not hardcoded UI copy.
